# TianXianQuant v1.0.0 Release Blockers

Last verified: 2026-05-16
Verification command: `bash scripts/verify_paid_release_config.sh`
Target build: TianXianQuant Android v1.0.0 (versionCode 1), AGP 8.6.1, Kotlin 1.9.22, compileSdk 35.

This document captures the **current output** of the two-stage release gate (bash precheck + Gradle `verifyPaidReleaseConfig`) when the user has not yet filled in production values, and explains what each failure means so it can be cleared one item at a time.

## Stage A — `scripts/check_paid_release_inputs.sh` (bash precheck)

The bash precheck is the first gate the wrapper runs. It rejects placeholder hosts (`example.com`, `your-domain`, `localhost`, `10.0.2.2`, etc.) and malformed URLs/emails **before** Gradle starts, so example values do not slip through into `BuildConfig`.

### Run 1: empty environment (no `release.env` sourced)

```text
Paid release input precheck passed.
```

Empty values pass the bash precheck because the precheck only validates non-empty fields. The Gradle gate (Stage B) is what enforces presence. So an "all empty" run jumps straight to Stage B.

### Run 2: literal placeholders from `release.env.example`

```text
Paid release inputs failed precheck:
 - TIANXIAN_PRODUCTION_API_BASE_URL uses a placeholder/local host
 - TIANXIAN_PRIVACY_POLICY_URL uses a placeholder/local host
 - TIANXIAN_TERMS_URL uses a placeholder/local host
 - TIANXIAN_DATA_DISCLAIMER_URL uses a placeholder/local host
 - TIANXIAN_SUPPORT_EMAIL uses a placeholder domain

Replace placeholder/example values with real production resources before running the Gradle gate.
```

**What this means.** Copying `release.env.example` to `release.env` and pasting the literal `example.com` / `support@example.com` values will fail this gate. This is intentional — `scripts/check_paid_release_inputs.sh` blocks placeholder hosts so the example values can't accidentally get baked into a signed AAB.

**How to clear.** Replace each value in `release.env` with a real production resource. See `docs/RELEASE_USER_CHECKLIST.md` Step 3.

## Stage B — `:app:verifyPaidReleaseConfig` (Gradle task)

If Stage A passes (or values are simply empty rather than placeholder-shaped), Gradle runs the paid-release config gate defined in `TianXianQuant/app/build.gradle.kts` (line 145). Captured output with an empty environment:

```text
> Task :app:verifyPaidReleaseConfig FAILED

FAILURE: Build failed with an exception.

* What went wrong:
Execution failed for task ':app:verifyPaidReleaseConfig'.
> Paid release configuration is incomplete:
   - -PtianxianApiBaseUrl=https://<deployed-production-api>/
   - -PtianxianPrivacyPolicyUrl=https://<public-privacy-policy>
   - -PtianxianTermsUrl=https://<public-terms>
   - -PtianxianDataDisclaimerUrl=https://<public-data-disclaimer>
   - -PtianxianSupportEmail=support@example.com
   - release signing properties outside git
```

Note: `tianxianBackendSyncEnabled=true` and `tianxianRequireBackendPaymentSync=true` are already injected by `scripts/verify_paid_release_config.sh`, so those two slots do not appear as missing.

### Required, currently failing

| # | Failure line | Source of truth | User action |
| --- | --- | --- | --- |
| 1 | `-PtianxianApiBaseUrl=https://<deployed-production-api>/` | Production backend deployed via `render.yaml` (or your own host) | Deploy backend; export `TIANXIAN_PRODUCTION_API_BASE_URL=https://<your-api-host>/` (must end with `/`, must not be `10.0.2.2`, must not be `api.gutongwealth.com`) |
| 2 | `-PtianxianPrivacyPolicyUrl=https://<public-privacy-policy>` | `docs/legal/PRIVACY_POLICY.md` hosted publicly | Enable GitHub Pages workflow (see RELEASE_USER_CHECKLIST Step 5) and export the public URL |

| 4 | `-PtianxianDataDisclaimerUrl=https://<public-data-disclaimer>` | `docs/legal/DATA_SOURCE_DISCLAIMER.md` hosted publicly | Same as #2 |
| 5 | `-PtianxianSupportEmail=support@example.com` | Operating entity mailbox | Provision a real mailbox (e.g. `support@tianxianquant.cn`), export `TIANXIAN_SUPPORT_EMAIL=...`, and update the placeholder in all three legal docs |
| 6 | `release signing properties outside git` | An upload keystore generated by the user | Run `keytool -genkey ...`, store keystore outside repo, export `TIANXIAN_RELEASE_*` (see RELEASE_USER_CHECKLIST Step 2) |

### Conditional failures (depend on the file the user provides)

| Condition | Failure line | Cause |
| --- | --- | --- |
| `TIANXIAN_RELEASE_KEYSTORE` path does not exist | `release signing keystore must point to an existing keystore file` | `keytool -genkey` was not run or `.jks` was moved |
| `TIANXIAN_RELEASE_KEYSTORE` set but one of store/alias/password env vars missing | `release signing store password, key alias, and key password must be provided via wrapper environment variables` | Partial `release.env`; all four `TIANXIAN_RELEASE_*` values must be set together |

## What's already cleared (no action needed)

These items would normally be additional gates, but the wrapper and `app/build.gradle.kts` already inject or enforce them correctly:

- Backend sync enabled (`-PtianxianBackendSyncEnabled=true`) — injected by wrapper.
- Backend payment sync required (`-PtianxianRequireBackendPaymentSync=true`) — injected by wrapper.
- Release signing values never pass through `-P` argv (the wrapper translates `TIANXIAN_RELEASE_*` to `ORG_GRADLE_PROJECT_tianxianRelease*` env vars and redacts them from output via the Python redactor at the tail of the wrapper).
- `.gitignore` already excludes `*.jks`, `*.keystore`, `*.p12`, `release.env`, `release-signing.properties`, and `TianXianQuant/release-signing.properties`.

## Net summary

There are **6 outstanding gate items**, all of which require user-supplied secrets or hosted resources that this preparation pass cannot produce on the user's behalf. Once those are in place the gate is expected to pass on the first attempt. See `docs/RELEASE_USER_CHECKLIST.md` for the precise sequence.
